The Multi-Robot Coordination Challenge

The Setup

  • Dual-arm robotic workcells with overlapping workspaces
  • Multiple robots must work simultaneously for throughput
  • Shared workspace creates collision risk

Key Challenge

How do we achieve concurrent motion while guaranteeing safety?
SAFETY "Zero collisions"
← TENSION →
THROUGHPUT "Maximize PPH"

Naive solution: one robot at a time (safe but ~40% slower)

  • Picture two robot arms on an assembly line - they frequently need to reach into the same workspace
  • The naive solution is sequential execution: Robot A completes, then Robot B starts. Safe, but we measured ~40% throughput loss
  • A collision is catastrophic: hardware damage ($50K+), production halt, safety incidents
  • The core question: can we prove mathematically that two trajectories won't collide, and if so, run them concurrently?
  • Technical context: Each robot has 6-7 DOF, trajectories are 2-8 seconds, collision checks must be under 1ms

"This isn't just an engineering problem - it's a business problem. Every second a robot waits is money lost."

Timing: 1.5 minutes

Requirements & Starting Point

Requirements

  • Zero tolerance to collisions - especially between the two robots
  • Real-time response - handle unhappy path scenarios (trajectory deviations, scene changes like a box falling, etc.)
  • Improve dual robot utilization - positively impacting system throughput

Existing Infrastructure

  • Collision checking - FCL-based geometry checks
  • Motion planning - OMPL path planner
  • Velocity profiling - TOPPRA time parameterization
  • Inugami runtime - C++ real-time controller
Initial Coordination: Bare-bones "spatial lock" - each robot acquired exclusive access to entire pick/place region, effectively serializing the arms.
  • Zero collisions is non-negotiable - we treat this as a hard constraint, not an optimization target
  • Real-time means decisions within 10ms for the control loop - we use async planning to avoid blocking
  • We already had solid per-robot infrastructure: collision checking with FCL, OMPL for path planning, TOPPRA for velocity profiling
  • Inugami is our C++ robot control runtime - handles real-time servo control, trajectory execution, and safety monitoring
  • The key gap: all these components worked independently per robot - there was no component that explicitly coordinated two robots or made them aware of each other

"We designed for the hard requirements first, then optimized for throughput."

Design Philosophy & Core Insight

Guiding Principles

  1. Separation of Concerns - Planning vs Execution vs Monitoring
  2. Temporal Awareness - Time is a first-class dimension
  3. Fail-Safe Behavior - When in doubt, stop safely

The Core Insight

SPATIAL-ONLY

"Will these paths EVER occupy the same space?"

→ If yes, block entirely

TEMPORAL

"Will they occupy the same space AT THE SAME TIME?"

→ If no, proceed concurrently

Like traffic intersections: cars cross the same point, just not simultaneously.

  • The key realization: two robots can traverse the same point in space, as long as they don't do it at the same time
  • This seems obvious in hindsight, but implementing it correctly requires careful handling of timing and validation
  • Spatial-only approaches block the entire "swept volume" for the full trajectory duration
  • Our temporal approach uses TOPPRA time parameterization to know exactly where each robot will be at any given time
  • We sample trajectories at 0.1s intervals and compare: "At t=2.3s, where is Robot A? Where is Robot B? Do they overlap?"

"We went from blocking space to blocking space-TIME. Time is now a first-class dimension."

Timing: 1.5 minutes

High-Level Architecture

Application: What to do (pick object A, place at B)
Planning: How to do it safely (paths, trajectories, coordination)
Control: Execute reliably (real-time control, safety monitoring)
  • Three-layer architecture with clear separation of concerns
  • Planning layer is Python for rapid iteration - we can update planning logic without recompiling the runtime
  • Core services are C++ for real-time guarantees - the CollisionMonitor runs at 100Hz with hard timing requirements
  • Key components:
    • CollisionScene: Maintains world model with robot meshes, obstacles, and active swept volumes
    • PathPlanner: Uses OMPL (Open Motion Planning Library) for sampling-based planning
    • TrajectoryGenerator: TOPPRA algorithm for time-optimal trajectory generation
    • TemporalValidator: Our innovation - validates trajectories considering time, not just space
  • Communication: Planning → Runtime uses gRPC with protobuf; Runtime → Planning uses async callbacks for progress updates

"Each layer can fail independently without compromising safety."

Trajectory Swept Volume: Time Parameterization

From Path to Timed Trajectory

Input: q₀ → q₁ → q₂ → ... → qₙ
Output: t=0.0s: q₀ | t=0.3s: q₁ | t=0.7s: q₂ | ... | t=N: qₙ

TrajectorySweptVolume: Time-Based Collision Entities

class TrajectorySweptVolume:
    def __init__(self, robot_collision_set, trajectory_data, time_step=0.1):
        # Sample at fixed time intervals using TOPPRA's reparameterized spline
        time_splits = [0.0, 0.1, 0.2, ..., duration]

        for t in time_splits:
            # Get exact joint angles at time t from TOPPRA spline
            joint_angles = trajectory_data.continuous_path(t)
            # Compute collision geometry at this configuration
            entities[t] = robot_collision_set.collision_entities(joint_angles)

This gives us: dict[time_seconds, list[CollisionEntity]]

  • TOPPRA (Time-Optimal Path Parameterization based on Reachability Analysis) is the key enabler
  • Input: geometric path (sequence of joint configurations) with no timing
  • Output: time-optimal trajectory respecting velocity (2 rad/s) and acceleration limits (5 rad/s²)
  • The reparameterized_spline gives us a continuous function: time → joint angles
  • We sample this at 0.1s intervals (configurable via TEMPORAL_ORCHESTRATION_TIME_STEP)
  • At each time sample, we compute the robot's collision geometry using forward kinematics
  • Result: a dictionary mapping absolute time (seconds) to collision entities
  • This is fundamentally different from progress-based sampling - we know exactly where the robot is at each moment in time

"TOPPRA gives us the time dimension. We know where the robot will be at t=2.3 seconds."

Temporal Collision Validation

This algorithm determines how long robot A should wait before starting its trajectory, given that robot B is currently executing a time-parameterized trajectory.

The Algorithm

def collision_check_trajectories_based_on_expected_states_at_time(
    robot_a_entities_at_time: dict[float, list[Entity]],  # {t: entities}
    robot_b_entities_at_time: dict[float, list[Entity]],  # {t: entities}
    wait_time: float = 0.0,
    lookahead_safety_time: float = 0.2,  # 200ms buffer
):
    for t_a, entities_a in robot_a_entities_at_time.items():
        for t_b, entities_b in robot_b_entities_at_time.items():
            # KEY: Only check if Robot B will still be at this position
            # when Robot A arrives (accounting for wait_time offset)
            if t_b <= (t_a + wait_time - lookahead_safety_time):
                continue  # Robot B will have moved past this point

            if collision(entities_a, entities_b):
                return COLLISION

    return SAFE

Wait Time Calculation

If collision detected, increment wait_time by 0.1s and retry until:

  • No collision → Robot A waits, then proceeds
  • wait_time > 5s → Fall back to sequential execution
  • This is the core temporal validation algorithm
  • For each time sample in Robot A's trajectory, we check against Robot B's remaining trajectory
  • The key insight: if t_b <= t_a + wait_time, Robot B will have already passed that point when Robot A arrives
  • The 200ms lookahead_safety_time adds a safety buffer for timing uncertainty
  • Wait time calculation: if there's a collision at t=2s, we try wait_time=0.1s, 0.2s, etc.
  • Maximum wait time of 5s (TEMPORAL_ORCHESTRATION_MAX_WAIT_TIME) - beyond this, we fall back to blocking
  • The function returns the required wait time, which is then applied before committing Robot A's trajectory
  • Constants from code: TEMPORAL_ORCHESTRATION_LOOKAHEAD_SAFETY_TIME = 0.2, TEMPORAL_ORCHESTRATION_TIME_STEP = 0.1

"We compute exactly how long Robot A needs to wait for Robot B to clear the overlap zone."

Collision Monitor - Safety Layer

C++ CollisionMonitor: Real-Time Swept Volume Checking

// From collision_monitor.cc - runs at 100Hz
CollisionCheckResult CheckCollisionFree(double current_progress, double current_speed) {
    // Compute velocity-based lookahead (400ms ahead)
    LookaheadResult lookahead = ComputeLookahead(current_progress, current_speed);

    // Update own swept volume window [current, current + lookahead]
    UpdateSelfSweptVolumeWindow(current_progress, lookahead.safe_progress);

    // Check against partner's swept volume (also windowed by their progress)
    collision_free = swept_volume_set_->IsCollisionFree(*other_swept_set);
}
  • Defense in depth: we don't trust any single layer to guarantee safety
  • Planning layer (Python): PathPlanner validates collision-free paths; TemporalValidator validates multi-robot timing
  • Runtime layer (C++): CollisionMonitor runs at 100Hz in a real-time thread (SCHED_FIFO priority)
  • The C++ CollisionMonitor uses the same swept volume concept but with velocity-based lookahead:
    • kLookaheadTime = 0.4 seconds ahead
    • Computes lookahead_distance = current_speed * 0.4s + safety_buffer
    • Uses MultiPoseArticulatedBodyCollisionSet for efficient windowed collision checking
  • Key optimization: UpdateWindow() takes ~1-2µs vs ~275µs for rebuilding the collision set
  • If collision predicted: triggers controlled deceleration to safe stop
  • Hardware E-Stop: Physical button, completely independent of software, cuts motor power

"The CollisionMonitor doesn't trust the planner. It verifies independently at 100Hz using the same temporal swept volume concept."

Additional Challenges & Solutions

Challenge 1: Trajectory Segment Handling

Problem: Trajectories have ingress/keypoints/egress segments with different safety needs

Solution: Split trajectory swept volume by segment, validate each separately

ingress_dict, keypoints_dict, egress_dict = split_trajectory_swept_volume_by_segments(
    trajectory_swept_volume,
    ingress_joint_path, keypoints_joint_path, egress_joint_path
)
# Check each segment with appropriate collision scenes

Challenge 2: Deadlock Prevention

Problem: Both robots could end up waiting for each other indefinitely

Solution: Maximum wait time threshold (5s) - if exceeded, fall back to sequential execution to guarantee progress

# If wait_time exceeds threshold, fall back to safe sequential mode
if wait_time > TEMPORAL_ORCHESTRATION_MAX_WAIT_TIME:  # 5.0 seconds
    return FALLBACK_TO_SEQUENTIAL
  • Segment handling: SE3 trajectories have three parts:
    • Ingress: approach to workspace (checked with deflated payload)
    • Keypoints: main manipulation (full collision checking)
    • Egress: retreat from workspace (checked with deflated payload)
    split_trajectory_swept_volume_by_segments() uses path distance fractions to divide the time-split volumes
  • Deadlock prevention: If temporal validation cannot find a valid wait time within 5 seconds, we fall back to the original spatial lock approach - safe but slower
  • Feature flag: commit_with_temporal_orchestration on trajectory strategy enables/disables per-trajectory

"The algorithm was 20% of the work. The edge cases were 80%."

Results & Impact

Performance Metrics

Metric Before (Sequential) After (Temporal) Improvement
System Throughput Baseline +40% +40%
Robot Wait Time Baseline -60% -60%
BEFORE: Robots took turns → 40% of potential throughput lost
AFTER: Temporal coordination → Improved dual robot utilization
At scale: 40% more picks/day = significant revenue impact

Video Demo: Dual-arm coordination with temporal orchestration

  • The headline number: 40% improvement in system throughput compared to sequential execution
  • This is measured as Picks Per Hour (PPH) in production workloads
  • Wait time reduction of 60% means robots spend much more time doing useful work
  • Zero collisions maintained - the temporal validation plus runtime CollisionMonitor catch all issues
  • Why "only" 40% and not 100%?
    • Some operations genuinely require exclusive access (both robots need same exact spot)
    • Task dependencies (Robot B needs Robot A to place something first)
    • Maximum 5s wait time before falling back to sequential
    • Theoretical maximum with perfect scheduling would be ~50-55% improvement
  • The remaining gap is in task-level scheduling, not motion coordination

"40% more throughput with zero hardware changes. Pure software improvement."
